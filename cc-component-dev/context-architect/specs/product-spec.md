## 一、产品定位

```
┌─────────────────────────────────────────────────────────────────────┐
│  Context Architect - Claude Code 上下文管理的智能工程化平台          │
├─────────────────────────────────────────────────────────────────────┤
│  愿景：让每个项目都拥有最优的 Claude Code 上下文配置                 │
│  定位：不是配置生成器，而是上下文健康管理系统                        │
│  差异化：系统性解决方案 vs 市场上的点状工具                          │
└─────────────────────────────────────────────────────────────────────┘
```

**Slogan 候选**：
- "Your Project, Perfectly Understood"
- "从 60% 到 90%+ 的 AI 理解准确率"

### 2.2 关键洞察

1. **核心价值 ≠ "生成配置"**
   核心价值 = "诊断问题 + 持续优化 + 智能维护"

2. **这是一个「上下文健康管理系统」而非「配置生成器」**

3. **效果驱动增长**：产品必须"开箱即用"就能展现价值

4. **数据是护城河**：通过积累项目类型最佳实践建立竞争壁垒

## 三、核心价值主张

| 痛点 | 解决方案 | 效果量化 |
|------|----------|----------|
| 不知道怎么写好 CLAUDE.md | 智能模板推荐 | 开箱即用的最佳实践 |

| 不同项目需要不同策略 | 自动项目类型识别 | 精准适配 |

## 五、MVP 功能规范

### 5.1 核心功能：智能项目适配

```
用户执行 /context-architect
        ↓
┌───────────────────────────────────────┐
│ 1. 项目扫描与类型识别                   │
│    - 分析 package.json, pyproject.toml │
│    - 识别框架、语言、架构模式            │
│    - 检测现有 CLAUDE.md 配置            │
└───────────────────────────────────────┘
        ↓
┌───────────────────────────────────────┐
│ 2. 最佳实践匹配                         │
│    - 从知识库匹配项目类型                │
│    - 加载对应的配置模板                  │
│    - 考虑项目规模和复杂度                │
└───────────────────────────────────────┘
        ↓
┌───────────────────────────────────────┐
│ 3. 配置生成与优化建议                   │
│    - 生成/优化 CLAUDE.md                │
│    - 提供改进建议和解释                  │
│    - 支持用户交互式调整                  │
└───────────────────────────────────────┘
```

### 5.2 MVP 支持的项目类型

**Phase 1 - Web 框架**

| 类型 | 框架示例 | 配置重点 |
|------|----------|----------|
| React SPA | React, CRA, Vite | 组件架构、状态管理、测试策略 |
| Next.js | Next.js 13/14/15 | App Router、SSR/SSG、API Routes |
| Vue | Vue 3, Nuxt | Composition API、Pinia、目录结构 |
| Python Web | FastAPI, Django | DDD 架构、ORM、API 设计 |
| Node Backend | Express, NestJS | 中间件、依赖注入、错误处理 |

**Phase 1 - 基础设施**

| 类型 | 工具示例 | 配置重点 |
|------|----------|----------|
| IaC | Terraform, CDK | 模块结构、状态管理、部署流程 |
| Kubernetes | K8s, Helm | 资源配置、部署策略、监控集成 |
| CI/CD | GitHub Actions | 工作流设计、环境管理 |

### 5.3 命令体系

```bash
/context-architect              # 主命令：智能分析 + 配置生成
/context-architect analyze      # 分析当前项目配置
/context-architect optimize     # 优化现有配置
/context-architect templates    # 浏览可用模板
/context-architect health       # 配置健康度检查 (Phase 2)
```

## 九、数据策略

### 9.1 数据收集范围

| 数据类型 | 收集方式 | 用途 |
|----------|----------|------|
| 项目类型分布 | 匿名聚合 | 优化模板推荐算法 |
| 配置模式 | 脱敏统计 | 发现最佳实践 |
| 用户反馈 | 主动提交 | 改进模板质量 |

### 9.2 隐私保护原则

1. **最小收集**：只收集改进产品必需的数据
2. **匿名化**：所有数据脱敏后再聚合
3. **用户控制**：用户可选择是否贡献数据
4. **透明度**：清晰说明数据用途

### 9.3 最佳实践数据来源

1. **自主整理**：团队总结的最佳实践
2. **GitHub 分析**：分析优秀开源项目的 CLAUDE.md
3. **社区贡献**：用户分享的成功配置案例
4. **迭代学习**：从用户反馈中持续优化
