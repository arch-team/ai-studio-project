# 训练管理模块 E2E 全面回归验证设计

> 日期：2026-06-12
> 目标平台：http://ai-platform-dev-alb-1343863355.us-east-1.elb.amazonaws.com/
> 当前部署版本：backend v1.2.20 / frontend v1.0.15

## 背景

今天早些时候已完成一轮训练管理模块深度 E2E 测试（57 个用例，修复 6 个 Bug，含暂停幂等、契约对齐、检查点端点补全等），修复已部署至 dev 集群。本轮为**全面回归验证**：重新执行全部既有测试套件，确认当前线上状态健康，仅对失败项排查修复，不重复造轮子。

## 范围

**纳入回归的测试套件**（frontend/e2e/tests/，共 57 个训练相关用例，以 `npx playwright test --list` 实际计数为准）：

| 套件 | 用例数 | 模式 |
|------|--------|------|
| training-jobs.spec.ts（列表） | 8 | Mock |
| training-jobs-crud.spec.ts（创建/详情/导航） | 21 | Mock |
| training-jobs-operations.spec.ts（暂停/恢复/删除/状态按钮矩阵） | 21 | Mock |
| training-real-smoke.spec.ts（真实集群冒烟，含暂停→恢复闭环） | 7 | 真实远程 |

**真实集群测试包含在内**：会在 dev 集群创建/删除真实 HyperPod CR，测试自带环境清理。只有真实链路能发现 DB 与 K8s 状态漂移类 Bug。

**不在范围**：开发空间（spaces）、模型、数据集等其他模块的测试套件；新增探索性用例（除非修复 Bug 需要 TDD 补测试）。

## 执行流程（方案 A：分层串行回归）

```
1. 预检    → 平台健康检查（前端 200、后端 API 可达）、kubectl 集群可达、
             种子数据状态确认（见下方种子任务清单）
2. Mock 层 → 串行跑 50 个 Mock 用例（training-jobs / crud / operations）
3. 真实层  → 跑 7 个真实集群冒烟用例（training-real-smoke）
4. 修复    → 对失败项逐个根因分析并修复（见下节）
5. 加固    → 对已知抖动用例（"failed" 状态按钮可见性）顺手加固：
             增加状态等待、数据隔离等，降低后续抖动概率
6. 终验    → 全量重跑确认；非抖动用例须一次通过，抖动用例允许单独重跑
             稳定通过（前提是已加固）；恢复集群环境（清理测试 CR、
             种子任务回归 3 个且状态正确）
```

**种子任务基线**（预检与终验的环境一致性标准）：

| ID | job_name | 期望状态 | 备注 |
|----|----------|---------|------|
| 1 | llama2-finetune-001 | completed | 列表/详情/检查点用例依赖 |
| 2 | sd-finetune-001 | completed | — |
| 3 | bert-pretrain-001 | running | 暂停→恢复闭环用例的前置条件 |

分层执行的理由：Mock 层失败 = 前端/契约问题，真实层失败 = 链路/集群问题，分层可以快速定位失败属于哪一类，与上轮已验证的工作方式一致。

## Bug 修复流程

发现失败时按以下流程处理，**禁止跳过测试或伪造结果**：

1. **根因分析**：区分真实 Bug / 环境抖动 / 测试自身问题
2. **TDD 修复**：先写失败的单元测试复现根因，再修代码使其通过
3. **质量门禁**：black + ruff + 相关单元测试全绿（前端为 lint + vitest）
4. **部署**：如需后端/前端改动，构建新镜像推 ECR（注意 tag 不可变，需递增版本号）并滚动部署到 EKS
5. **真实环境复验**：部署后在真实平台重跑相关 E2E 用例确认修复生效

## 验收标准

- 全部 57 个用例通过：非抖动用例在终验全量运行中一次通过；抖动用例允许单独重跑稳定通过，但必须已加固（如增加状态等待、隔离数据）
- 集群环境恢复干净：无残留测试 CR（dev 集群无 GPU 节点，残留 CR 会堵塞调度队列），种子任务符合上方基线表
- 所有修复（如有）已提交 git 并部署生效，且经真实环境复验
