# 计划: 软件产品全生命周期研发流程管理框架

## Context

当前 ai-studio-project 已构建了成熟的 Spec-Kit 规范驱动开发体系（9 个命令覆盖从治理到实施），但生命周期的**后半段偏弱**——实施完成之后缺少结构化的验收验证、代码审查自动化、发布管理、回顾复盘和技术债务管理。同时拥有大量扩展组件（17 个 Agent、9 个 Plugin、8+ MCP Server、26 个 SC 命令、7 个行为模式）但缺少将它们串联为完整研发流水线的编排层。

**目标**: 创建一套完整的生命周期编排框架，使 Claude Code 具备类似人类团队的软件研发流程管理能力——从需求到发布、从回顾到知识沉淀的完整闭环。

## 编排设计: 全流程连接

### 主流程 (Happy Path)

```
/speckit.constitution ──► /speckit.specify ──► [/speckit.clarify] ──► /speckit.plan
                                                                           │
/speckit.checklist ◄── /speckit.analyze ◄── /speckit.tasks ◄───────────────┘
       │
       ▼
/speckit.implement ──── (Agent Teams 波次执行)
       │
       ▼
/lifecycle.review ──── (多维度审查: 架构/代码/安全/测试)
       │ FAIL → 修复循环
       ▼ PASS
/lifecycle.test-gate ──── (测试门禁: 覆盖率 + Evals 汇总)
       │ FAIL → 修复 → 重新检查
       ▼ PASS
/lifecycle.accept ──── (验收: spec 验收标准 × tasks 完成率 × checklist)
       │ 未通过 → 补充实现
       ▼ 通过
/lifecycle.release ──── (发布: CHANGELOG + Tag + PR)
       │
       ▼
/lifecycle.retro ──── (回顾: KPT + 量化指标)
       │
       ├── 发现模式 → /lifecycle.knowledge (提取 Skill)
       └── 发现债务 → /lifecycle.debt (记录追踪)
```

### 辅助流程 (触发式)
- **规范变更时** → `/lifecycle.impact` (增量影响分析)
- **Phase 结束时** → `/lifecycle.retro` (回顾复盘)
- **每周/每 Phase** → `/lifecycle.debt scan` (技术债务扫描)
- **好模式出现时** → `/lifecycle.knowledge skill` (提取 Skill)

## 各 Command 核心设计

### `/lifecycle.review` (P7 代码审查)

```
输入: 子项目 (backend|frontend|cdk) + 范围 (模块名)
流程:
  1. 读取对应 checklist.md (已有 3 套)
  2. 运行四维度审查:
     - 架构合规 → test_architecture_compliance.py / ESLint / CDK Nag
     - 代码质量 → black + ruff + mypy / tsc + eslint / ruff + mypy
     - 安全检查 → 引用 security.md 规范
     - 测试质量 → 覆盖率 + 边界用例
  3. 输出结构化审查报告
  4. handoff → /lifecycle.test-gate 或修复循环
复用: agent-teams-guide.md 模式 C、3 套 checklist.md
```

### `/lifecycle.test-gate` (P8 测试门禁)

```
输入: 子项目 + 目标覆盖率 (可选)
流程:
  1. 运行测试:
     后端: pytest --cov=src --cov-report=json
     前端: npm test -- --run --coverage
     CDK:  pytest -m unit --cov=stacks --cov=cdk_constructs
  2. 覆盖率 vs constitution.md 要求 (后端≥85%, 前端≥80%, CDK stacks≥90%)
  3. 汇总 .claude/evals/ 中的 Eval 通过状态
  4. 判定: PASS / WARN / FAIL
  5. handoff → /lifecycle.accept (PASS) 或修复循环 (FAIL)
```

### `/lifecycle.accept` (P9 验收确认)

```
输入: feature 目录 (specs/{feature}/)
流程:
  1. 读取 spec.md User Stories + 验收标准
  2. 统计 tasks.md 完成率 ([X] 数 / 总数)
  3. 汇总 checklists/ 状态
  4. 内部调用 /lifecycle.test-gate
  5. 输出验收矩阵:
     | User Story | Tasks | Tests | Checklist | 判定 |
  6. handoff → /lifecycle.release (全部通过)
```

### `/lifecycle.release` (P10 发布管理)

```
输入: 版本号 + 类型 (major|minor|patch)
流程:
  1. 前置检查: /lifecycle.accept 已通过
  2. 生成 CHANGELOG (git log 按 commit 类型分组)
  3. 创建 Git Tag
  4. gh pr create (body 含验收摘要 + CHANGELOG + 覆盖率)
  5. CDK 额外: cdk diff 输出变更影响
```

### `/lifecycle.impact` (P11 变更影响)

```
输入: 变更文件 (spec.md|plan.md|constitution.md|tasks.md)
流程:
  1. 检测变更类型和范围
  2. serena 查找引用变更实体的代码
  3. 交叉检查 tasks.md 已完成任务
  4. 输出影响矩阵 (变更项 × 影响范围 × 风险等级)
  5. 可选: 生成增量任务追加到 tasks.md
```

### `/lifecycle.retro` (P12 回顾复盘)

```
输入: 范围 (sprint|phase|incident) + 时间范围
流程:
  1. 收集: serena memory + cc-doc/plans/ + git log
  2. sequential-thinking 结构化分析
  3. 输出 KPT 报告 (Keep / Problem / Try)
  4. 知识提取触发:
     成功模式 → 建议 /lifecycle.knowledge skill
     问题根因 → 建议更新 .claude/rules/
     行动项 → 建议 /lifecycle.debt add
```

### `/lifecycle.debt` (P13 技术债务)

```
输入: 操作 (scan|add|list|prioritize|resolve) + 子项目
  scan: Grep TODO/FIXME/HACK + 低覆盖率模块 + lint 坏味道 + 过旧依赖
  add: 交互式记录 → 写入 debt-register.md
  list: 汇总全局债务看板
  prioritize: sequential-thinking 分析影响/紧迫度
  resolve: 选择债务项 → 生成任务 → 追加 tasks.md
```

### `/lifecycle.knowledge` (P14 知识管理)

```
输入: 操作 (extract|skill|guide|audit)
  extract: 从 serena memory + plans/ 提取候选知识
  skill: 以 decorator-exception SKILL.md 为模板生成新 Skill
  guide: 以 agent-teams-guide.md 为模板生成新指南
  audit: 检查 rules/ 与代码对齐度 + CLAUDE.md 过时信息
```

## 验证方法

1. **单命令验证**: 每个新 Command 编写后，以当前项目的 US1 (训练任务管理) 为测试对象，手动执行验证输出格式和逻辑正确性
2. **Handoff 链路验证**: 执行 `/lifecycle.review` → `/lifecycle.test-gate` → `/lifecycle.accept` 完整链路，验证 handoff 衔接
3. **回归验证**: 新 Command 不能破坏现有 9 个 Spec-Kit 命令的功能
4. **端到端验证**: 以一个小型 feature 为例 (如 Phase 4 的 datasets 前端)，执行 `/speckit.implement` → 全部 lifecycle 命令链路
5. **已有 Evals 持续通过**: 4 个已通过的 Evals 在框架引入后仍需保持 PASS
