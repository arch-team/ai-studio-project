# 代码规范整合方案

## 目标

将项目中分散的代码规范内容按项目类型整合为**三个独立的 Claude Code Rules 文件**。

**设计决策**:
- ✅ 后端、前端、CDK 作为独立项目，各自维护独立的编码规范
- ✅ 行长度保持现状：后端 120 字符，CDK 88 字符
- ✅ 使用 Claude Code 官方 `.claude/rules/` 目录结构

---

## 现状分析

### 当前文件分布（16+ 个文件）

```
📁 代码规范相关文件
├── /CLAUDE.md                              # 全局入口 (语言规范、TDD、项目概述)
├── /backend/
│   ├── CLAUDE.md                           # 后端技术栈、行长度(120)、异常处理
│   ├── docs/ARCHITECTURE.md                # DDD 架构、黄金法则 R1-R4 (800+行)
│   ├── tests/CLAUDE.md                     # 测试命名、分层比例、Mock 策略
│   └── pyproject.toml                      # Black/Ruff/MyPy 配置
├── /frontend/
│   ├── CLAUDE.md                           # Feature-Sliced Design
│   └── DESIGN.md                           # Cloudscape-First、组件规范
├── /infrastructure/cdk/
│   ├── CLAUDE.md                           # Stack 分层、安全底线
│   ├── pyproject.toml                      # CDK 特有配置 (88字符)
│   └── .claude/rules/                      # 8 个按需加载规则
└── /specs/001-ai-training-platform/
    └── spec.md                             # 术语标准
```

### 问题分析

| 问题类型 | 具体问题 |
|---------|---------|
| **重复定义** | 行长度、TDD 流程在多处定义 |
| **分散管理** | 后端规范分布在 4 个文件中 |
| **格式不一致** | CDK 使用 rules/ 目录，后端/前端使用 CLAUDE.md |
| **维护困难** | 修改规范需要更新多个文件 |

---

## 推荐方案：三项目独立规范

### 整合后的文件结构

```
📁 项目根目录
├── CLAUDE.md                           # 简化：语言规范 + 项目概述 + 索引
├── backend/
│   ├── .claude/
│   │   └── rules/
│   │       └── code-standards.md       # 🆕 后端统一规范
│   ├── CLAUDE.md                       # 简化：指向 rules + 模块职责
│   └── docs/ARCHITECTURE.md            # 保留：详细架构参考
├── frontend/
│   ├── .claude/
│   │   └── rules/
│   │       └── code-standards.md       # 🆕 前端统一规范
│   └── CLAUDE.md                       # 简化：指向 rules
└── infrastructure/cdk/
    └── .claude/
        └── rules/
            └── code-standards.md       # 🆕 CDK 统一规范 (整合现有 8 文件)
```

## 三个规范文件详细设计

### 1. 后端规范：`backend/.claude/rules/code-standards.md`

**来源整合**:
- `backend/CLAUDE.md` → 技术栈、异常处理、AWS 异步
- `backend/docs/ARCHITECTURE.md` → DDD 四层、黄金法则 R1-R4
- `backend/tests/CLAUDE.md` → 测试命名、分层比例
- `backend/pyproject.toml` → 行长度 120、工具配置

**内容结构**:
```markdown
---
description: 后端 Python 代码规范 - FastAPI + DDD + Clean Architecture
alwaysApply: true
---

# 后端代码规范

## 1. 代码风格
- 行长度: 120 字符
- 缩进: 4 空格
- 命名: snake_case (方法/变量), PascalCase (类)
- 类型注解: 强制 (mypy strict)
- Docstring: 单行中文，禁止 Args/Returns

## 2. 架构规范 (DDD 四层)
- 依赖方向: API → Application → Domain ← Infrastructure
- 黄金法则 R1-R4
- ORM 外键例外

## 3. 异常处理
- @problem 装饰器 + @dataclass
- 域异常层次

## 4. AWS 异步操作
- 强制 aioboto3 原生异步
- 禁止 run_in_executor

## 5. 测试规范
- TDD 红绿重构
- 分层比例: Unit 70% / Integration 20% / E2E 10%
- 命名前缀: test_entity_, test_svc_, test_api_

## 6. 禁止事项
- 同步 boto3
- Domain 层跨模块导入
- Celery 后台任务
```

---

### 2. 前端规范：`frontend/.claude/rules/code-standards.md`

**来源整合**:
- `frontend/CLAUDE.md` → 技术栈、Feature-Sliced Design
- `frontend/DESIGN.md` → Cloudscape-First、组件规范、禁止事项

**内容结构**:
```markdown
---
description: 前端 TypeScript 代码规范 - React + Cloudscape + Feature-Sliced
alwaysApply: true
---

# 前端代码规范

## 1. 代码风格
- 组件命名: PascalCase
- 方法/变量: camelCase
- 缩进: 2 空格
- 类型: 强制 TypeScript strict

## 2. 架构规范 (Feature-Sliced Design)
- 模块隔离: 通过 index.ts 导入
- 状态分层: TanStack Query (服务器) + Zustand (客户端)
- 路径别名: @features/, @shared/, @lib/

## 3. UI 规范 (Cloudscape-First)
- 禁止自定义 CSS
- 禁止内联样式
- 禁止原生 HTML 元素
- 使用 Cloudscape 组件

## 4. 操作反馈时效
- 按钮点击: 即时 Loading
- 表单提交: ≤100ms
- API 请求: ≤3s 进度指示

## 5. 测试规范
- 分层比例: Unit 60% / Integration 25% / E2E 15%
- 组件测试: React Testing Library
- E2E: Playwright

## 6. 禁止事项
- 内联样式 style={{}}
- 自定义 CSS 文件
- 原生 <div>, <button>
- 硬编码颜色值
```

---

### 3. CDK 规范：`infrastructure/cdk/.claude/rules/code-standards.md`

**来源整合**:
- `infrastructure/cdk/CLAUDE.md` → Stack 分层、安全底线
- `infrastructure/cdk/.claude/rules/01-08` → 8 个规则文件合并
- `infrastructure/cdk/pyproject.toml` → 行长度 88

**内容结构**:
```markdown
---
description: CDK Python 代码规范 - AWS CDK + Stack 分层 + 安全规范
alwaysApply: true
---

# CDK 代码规范

## 1. 代码风格
- 行长度: 88 字符 (PEP8)
- Construct ID: PascalCase ("MainVpc")
- 资源名称: kebab-case ("ai-platform-dev-vpc")
- 私有方法: _snake_case
- 类型注解: 强制 (mypy strict)

## 2. Stack 分层架构
Layer 1: NetworkStack, IamStack (并行)
Layer 2: DatabaseStack, StorageStack (并行)
Layer 3: EksStack → SagemakerHyperPodStack → HyperPodAddonsStack
Layer 4: FsxLustreStack
Layer 5: AlbStack

## 3. 依赖规则
- 上层只依赖下层
- 构造函数参数传递资源
- 禁止 Fn.import_value()

## 4. 安全规范
- 最小权限 IAM
- KMS 加密
- 私有子网
- IMDSv2
- CDK Nag 检查

## 5. 测试规范
- 分层比例: Unit 70% / Integration 30%
- CDK assertions 验证

## 6. 禁止事项
- Fn.import_value() 跨 Stack 引用
- 硬编码 ARN
- 裸 except 语句
- Any 类型
- 全局变量
```
