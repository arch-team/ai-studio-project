# API 响应格式标准优化计划

## 背景分析

### 当前系统现状

**好消息**：Domain 异常基类已定义 `error_code` 属性（`exceptions.py:31`），但异常处理器未使用它。

**问题**：存在 **4 种不同的错误响应格式**：

| 来源 | 格式 | 问题 |
|------|------|------|
| Domain 异常 | `{"detail": "...", "error_type": "EntityNotFoundError"}` | 使用类名而非 error_code |
| Security 异常 | `{"detail": "...", "error_code": "INVALID_CREDENTIALS"}` | 已有 error_code ✅ |
| Auth 中间件 | `{"code": "...", "message": "..."}` | 字段名不一致 |
| FastAPI 默认 | `{"detail": "..."}` | 缺少错误码 |

**前端期望**（`errors.ts:71-79`）：
```typescript
interface ApiErrorResponse {
  error: {
    code: string;
    message: string;
    details?: Record<string, unknown>;
    trace_id?: string;
  };
}
```

### 核心差距

1. **未使用 error_code**：Domain 异常处理器返回 `error_type`（类名），未读取 `exc.error_code`
2. **缺少包装**：后端直接返回字段，前端期望包装在 `error` 对象中
3. **字段名不一致**：`detail` vs `message`，`error_type` vs `code`

---

## 优化方案

### 目标格式（统一后）

```json
{
  "error": {
    "code": "TRAINING_JOB_NOT_FOUND",
    "message": "TrainingJob with id '123' not found",
    "details": {
      "entity_type": "TrainingJob",
      "entity_id": "123"
    },
    "trace_id": "req-abc123"
  }
}
```

## 实施计划

### Phase 1: 统一异常处理器输出（高收益，低工作量）

**目标**：让所有异常返回统一格式，使用已有的 `error_code` 属性

**文件**：`backend/src/shared/api/exception_handlers.py`

**改动**：

```python
# 改动前 (line 113-119)
return JSONResponse(
    status_code=status_code,
    content={
        "detail": exc.message,
        "error_type": type(exc).__name__,
    },
)

# 改动后
return JSONResponse(
    status_code=status_code,
    content={
        "error": {
            "code": exc.error_code,  # 使用已有属性
            "message": exc.message,
            "details": _get_error_details(exc),
            "trace_id": getattr(request.state, "trace_id", None),
        }
    },
)
```

同样修改 `security_exception_handler` 以返回包装格式。

### Phase 2: 修复中间件错误格式

**文件**：`backend/src/shared/api/middleware/auth.py`

**改动**：统一返回格式为 `{"error": {"code": "...", "message": "..."}}`

### Phase 3: 添加 trace_id 中间件

**新建文件**：`backend/src/shared/api/middleware/tracing.py`

```python
class TracingMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        trace_id = request.headers.get("X-Request-ID") or str(uuid.uuid4())[:8]
        request.state.trace_id = trace_id
        response = await call_next(request)
        response.headers["X-Request-ID"] = trace_id
        return response
```

### Phase 4: 增强 Domain 异常的 details 支持

**文件**：`backend/src/shared/domain/exceptions.py`

**改动**：添加 `get_details()` 方法

```python
class EntityNotFoundError(DomainError):
    def get_details(self) -> dict:
        return {"entity_type": self.entity_type, "entity_id": self.entity_id}
```

## RFC 9457 评估

| 方面 | 是否采用 | 理由 |
|------|---------|------|
| `type` URI | ❌ 不采用 | 维护成本高，`code` 字符串足够 |
| `title` | ❌ 不采用 | 与 `message` 重复 |
| `status` | ❌ 不采用 | HTTP 状态码已在响应头 |
| `detail` | ✅ 采用 | 重命名为 `message` |
| `instance` | ❌ 不采用 | `trace_id` 足够追踪 |
| 扩展字段 | ✅ 采用 | 使用 `details` 对象 |

**结论**：采用 **简化版 RFC 9457** + **Google API 风格**，平衡标准化与实现复杂度。

## 预期收益

1. **前端开发效率**：统一格式，减少条件判断
2. **调试效率**：trace_id 支持端到端追踪
3. **可维护性**：error_code 替代字符串匹配
4. **行业对齐**：接近 Google Cloud API / Stripe API 风格
