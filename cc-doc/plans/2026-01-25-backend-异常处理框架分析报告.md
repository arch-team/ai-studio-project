# Backend 异常处理框架分析报告

## 1. 执行摘要

**结论**: Backend 项目**存在双轨异常体系设计问题**，需要统一或明确职责分工。

| 评估维度 | 状态 | 说明 |
|---------|------|------|
| 统一基类 | ⚠️ 双轨并行 | `DomainError` + `SecurityError` 两套独立体系 |
| HTTP 映射 | ⚠️ 部分缺失 | auth 模块异常未映射到正确状态码 |
| 全局处理器 | ✅ 已注册 | `main.py` 正确注册 3 个处理器 |
| 模块继承 | ✅ 正确 | 所有模块异常正确继承基类 |
| 代码重复 | ❌ 需改进 | auth 模块与 security 层存在 11 个同名异常 |

### 3.2 核心问题：双轨异常体系 🔴

#### 问题描述

项目存在**两套独立的认证/授权异常体系**:

| 体系 | 位置 | 基类 | 异常数 |
|------|------|------|-------|
| **Auth 体系** | `modules/auth/domain/exceptions.py` | `AuthError(DomainError)` | 13 个 |
| **Security 体系** | `shared/infrastructure/security/exceptions.py` | `SecurityError(Exception)` | 12 个 |

#### 同名异常对比 (11 个重复)

| 异常名 | Auth 版本 (继承) | Security 版本 (继承) | HTTP 映射 |
|--------|-----------------|---------------------|-----------|
| `UserNotFoundError` | `AuthError` | `SecurityError` | 400 vs 404 |
| `InvalidCredentialsError` | `AuthError` | `SecurityError` | 400 vs 401 |
| `AccountLockedError` | `AuthError` | `SecurityError` | 400 vs 423 |
| `TokenExpiredError` | `TokenError(AuthError)` | `SecurityError` | 400 vs 401 |
| `InvalidTokenError` | `TokenError(AuthError)` | `SecurityError` | 400 vs 401 |
| `PasswordExpiredError` | `AuthError` | `SecurityError` | 400 vs 401 |
| `PasswordTooWeakError` | `AuthError` | `SecurityError` | 400 vs 400 |
| `PasswordHistoryViolationError` | `AuthError` | `SecurityError` | 400 vs 400 |
| `InsufficientPermissionsError` | `AuthError` | `SecurityError` | 400 vs 403 |
| `SSOError` | `AuthError` | `SecurityError` | 400 vs 401 |
| `SSODegradedModeError` | `SSOError(AuthError)` | `SSOError(SecurityError)` | 400 vs 503 |

#### 影响分析

**问题 1**: HTTP 状态码不一致
- Auth 模块异常继承 `DomainError`，被 `domain_exception_handler` 处理
- `DOMAIN_EXCEPTION_MAP` 中没有 auth 异常的映射，**默认返回 400**
- 例如：auth 模块的 `InvalidCredentialsError` 返回 400，而非期望的 401

**问题 2**: 代码维护风险
- 同名异常在不同地方定义，容易混淆
- 修改一处不会自动同步到另一处

#### Training 模块 ✅ (已正确实现)

HyperPod 异常**已正确继承** `TrainingError`:

```python
# training/domain/exceptions.py - 实际代码
class HyperPodSDKUnavailableError(TrainingError):  # ✅ 正确
class HyperPodPodNotFoundError(TrainingError):     # ✅ 正确
class HyperPodOperationError(TrainingError):       # ✅ 正确
```

**次要问题**: 这些异常会返回默认的 400，而更合适的映射可能是:
- `HyperPodSDKUnavailableError` → 503 (服务不可用)
- `HyperPodPodNotFoundError` → 404 (资源不存在)

## 5. 改进建议

### 建议 1: 统一 Auth/Security 异常体系 (推荐方案 B)

**方案 A**: 删除 auth 模块异常，直接使用 Security 层

```python
# auth/application/services.py
from src.shared.infrastructure.security.exceptions import (
    InvalidCredentialsError,  # 直接使用 security 版本
    TokenExpiredError,
    AccountLockedError,
)
```

- ✅ 简单直接，无重复代码
- ❌ 违反 DDD 模块隔离原则 (Domain 层不应依赖 Infrastructure)

**方案 B (推荐)**: Auth 模块使用 Security 异常作为基类

```python
# auth/domain/exceptions.py
from src.shared.infrastructure.security.exceptions import SecurityError

class AuthError(SecurityError):  # 改为继承 SecurityError
    """Auth 模块基础异常 - 继承 Security 异常以获得正确的 HTTP 映射"""
    pass

# 其余异常保持不变，继承 AuthError
class InvalidCredentialsError(AuthError): ...
```

- ✅ 保持模块独立性
- ✅ 自动获得 `security_exception_handler` 的正确 HTTP 映射
- ⚠️ 需要在 `SECURITY_EXCEPTION_MAP` 中添加 auth 异常的映射

**方案 C**: 在 `DOMAIN_EXCEPTION_MAP` 中添加 auth 异常映射

```python
# shared/api/exception_handlers.py
from src.modules.auth.domain.exceptions import (
    InvalidCredentialsError as AuthInvalidCredentialsError,
    TokenExpiredError as AuthTokenExpiredError,
    # ...
)

DOMAIN_EXCEPTION_MAP: dict[type[DomainError], int] = {
    # 现有映射...
    AuthInvalidCredentialsError: status.HTTP_401_UNAUTHORIZED,
    AuthTokenExpiredError: status.HTTP_401_UNAUTHORIZED,
    # ...
}
```

- ✅ 无需修改异常定义
- ❌ 维护两套映射，容易遗漏

### 建议 2: 添加 HyperPod 异常映射

```python
# shared/api/exception_handlers.py
from src.modules.training.domain.exceptions import (
    HyperPodSDKUnavailableError,
    HyperPodPodNotFoundError,

DOMAIN_EXCEPTION_MAP: dict[type[DomainError], int] = {
    # 现有映射...
    HyperPodSDKUnavailableError: status.HTTP_503_SERVICE_UNAVAILABLE,
    HyperPodPodNotFoundError: status.HTTP_404_NOT_FOUND,
```

### 建议 3: 添加异常元数据 (长期改进)

```python
# shared/domain/exceptions.py
class DomainError(Exception):
    """增强的基类 - 自带 HTTP 状态码"""
    http_status: int = 400
    error_code: str = "DOMAIN_ERROR"

    def __init__(self, message: str):
        self.message = message
        super().__init__(message)

class EntityNotFoundError(DomainError):
    http_status = 404
    error_code = "ENTITY_NOT_FOUND"
```

然后简化处理器:

```python
async def domain_exception_handler(request: Request, exc: DomainError) -> JSONResponse:
    return JSONResponse(
        status_code=exc.http_status,  # 直接从异常获取
        content={"detail": exc.message, "error_code": exc.error_code},
    )
```

## 8. 结论

### 总体评估

Backend 项目**有统一异常处理框架的意图**，但实际存在**双轨设计问题**:

| 方面 | 评估 | 说明 |
|------|------|------|
| 框架设计 | ⚠️ | 存在 DomainError 和 SecurityError 两套独立体系 |
| 模块遵循 | ✅ | 除 auth 外，其他模块正确继承 DomainError |
| HTTP 映射 | ⚠️ | auth 模块异常返回默认 400，而非正确状态码 |
| 代码重复 | ❌ | 11 个同名异常在两处定义 |

### 核心发现

1. **双轨异常体系是主要问题**
   - `DomainError` 体系用于业务模块 (training, quotas, spaces 等)
   - `SecurityError` 体系用于认证基础设施
   - `auth` 模块错误地使用了 `DomainError` 体系

2. **HTTP 状态码不一致**
   - auth 模块的 `InvalidCredentialsError` 返回 400（应为 401）
   - auth 模块的 `AccountLockedError` 返回 400（应为 423）
   - 这会导致前端错误处理不准确

3. **HyperPod 异常已正确实现**（修正初始分析）
   - 都正确继承了 `TrainingError(DomainError)`
   - 仅缺少专门的 HTTP 状态码映射

### 建议优先级

```
1. 🔴 统一 Auth/Security 异常体系 (方案 B 推荐)
   └→ 让 AuthError 继承 SecurityError
   └→ 在 SECURITY_EXCEPTION_MAP 添加映射

2. 🟡 添加 HyperPod 异常 HTTP 映射
   └→ HyperPodSDKUnavailableError → 503
   └→ HyperPodPodNotFoundError → 404

3. 🟢 长期改进：异常自带 http_status 属性
   └→ 减少映射表维护负担
```
