# 代码审查报告: AWS 异步操作规范违规分析

> **审查日期**: 2026-01-26
> **审查范围**: backend 项目中违反 `CLAUDE.md` AWS 异步操作规范的代码
> **严重级别**: 🔴 HIGH (违反强制规范)

---

## 规范要求 (来自 backend/CLAUDE.md)

```
规则: 所有 AWS S3/FSx 等需要异步的操作，**必须**使用 `aioboto3`，**禁止**自己封装 `run_in_executor`。

原因:
- `run_in_executor` 使用线程池，有额外开销
- aioboto3 提供真正的异步 I/O，性能更好
- 代码更简洁，减少重复的包装模式
```

## 详细分析

### 1. HyperPodClient (🔴 HIGH)

**文件**: `src/modules/training/infrastructure/hyperpod/client.py`

**违规代码位置**:
- L8: `import boto3`
- L80: `self._sagemaker_client = boto3.client("sagemaker", region_name=region)`
- L119-122: `_run_in_executor` 封装方法
- L552, L579: S3 操作使用同步 boto3.client("s3")

**违规模式**:
```python
# ❌ 当前实现 (违规)
import boto3

class HyperPodClient:
    def __init__(self, region: str = "us-east-1"):
        self._sagemaker_client = boto3.client("sagemaker", region_name=region)

    async def _run_in_executor(self, func: Callable[[], T]) -> T:
        loop = asyncio.get_running_loop()
        return await loop.run_in_executor(None, func)

    async def describe_cluster(self, cluster_name: str) -> dict[str, Any]:
        return await self._run_in_executor(
            lambda: self._sagemaker_client.describe_cluster(ClusterName=cluster_name)
        )
```

**问题分析**:
1. **SageMaker API 操作** (L201-242): `create_cluster`, `describe_cluster`, `list_clusters` 等使用 `boto3.client("sagemaker")` + `run_in_executor`
2. **S3 检查点操作** (L534-600): `verify_checkpoint_exists`, `list_checkpoints` 在闭包内创建 `boto3.client("s3")`
3. **HyperPod SDK 操作** (L325-788): `submit_training_job`, `get_training_job_status` 等虽然使用 HyperPod SDK（无 async 版本），但 S3 操作应使用 aioboto3

**受影响的 API 端点**:
- `POST /api/v1/training-jobs` (提交任务)
- `GET /api/v1/training-jobs/{id}/status` (状态查询)
- `POST /api/v1/training-jobs/{id}/stop` (停止任务)
- `GET /api/v1/clusters` (集群列表)

## 修复建议

### HyperPodClient 修复方案

**方案 A: 重构 SageMaker/S3 操作为 aioboto3** (推荐)

```python
import aioboto3

class HyperPodClient:
    def __init__(self, region: str = "us-east-1"):
        self._region = region
        self._session = aioboto3.Session()

    async def describe_cluster(self, cluster_name: str) -> dict[str, Any]:
        async with self._session.client("sagemaker", region_name=self._region) as sagemaker:
            return await sagemaker.describe_cluster(ClusterName=cluster_name)

    async def verify_checkpoint_exists(self, s3_path: str) -> bool:
        bucket, key = self._parse_s3_path(s3_path)
        async with self._session.client("s3", region_name=self._region) as s3:
            try:
                await s3.head_object(Bucket=bucket, Key=key)
                return True
            except ClientError:
                return False
```

**方案 B: 分离关注点**

将 S3 操作委托给已有的 `S3StorageClient`:
```python
class HyperPodClient:
    def __init__(self, region: str, s3_client: S3StorageClient):
        self._s3_client = s3_client

    async def verify_checkpoint_exists(self, s3_path: str) -> bool:
        return await self._s3_client.file_exists(s3_path)
```

### HyperPod SDK 操作的特殊处理

HyperPod SDK (`HyperPodPytorchJob`) 是同步 API，没有异步版本。对于这类操作:
1. `run_in_executor` 是**可接受的临时方案**
2. 应添加注释说明原因
3. 监控 AWS 官方异步 SDK 的发布

```python
# HyperPod SDK 无异步版本，使用 run_in_executor 包装是可接受的临时方案
# TODO: 当 sagemaker-hyperpod 提供 async API 时迁移
async def submit_training_job(self, ...):
    return await self._run_in_executor(_submit)
```

## 结论

backend 项目中存在 **2 处明确违规** (HyperPodClient 中的 boto3 + run_in_executor):

1. **SageMaker API 操作**: 应迁移到 aioboto3
2. **S3 检查点操作**: 应复用或参考 `S3StorageClient` 实现

项目已有 **正确的参考实现** (`s3_client.py`, `multipart_upload_client.py`)，修复工作量可控。

MLflowService 的 `run_in_executor` 是**可接受的例外**，因 MLflow SDK 无异步版本，但应添加注释说明。
